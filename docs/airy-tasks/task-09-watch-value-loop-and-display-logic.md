# Task 09 Draft — Agent Watch 价值感与展示逻辑升级

> 状态：规划草案，先用于 Claude 碰撞，不直接作为实现 spec。
>
> 目标：让用户停留在 `/agent` 页面时持续感受到价值，而不是每 3-5 分钟只看到三句 Agent 消息。
>
> 前置：Task 08 多币种池（固定 4 + 热门 3 + 机会 3）应先落地或同步进入同一规划链。

---

## 1. 问题定义

当前 Act 1.5 Watch 页面已经具备实时行情 + Agent 分析雏形，但体验仍像“定时吐三句话的聊天流”：

- 行情范围只有 4 个固定币，信息密度低。
- Agent 每轮只输出 Alpha/Beta/Gamma 三条，看完后页面进入等待。
- 用户无法快速理解“当前市场发生了什么变化”。
- Agent 的观点缺少持久承载，历史消息虽然存在，但不等于可读的观点面板。
- 页面没有把低成本可计算信号和高成本 LLM 解读分层，导致价值感过度依赖 LLM 刷新。

核心判断：`/agent` 不应该只是消息流，而应该是“AI 行情旁观台”。

---

## 2. 产品目标

用户停留在页面时，应该持续获得三类价值：

1. **市场热度感知**：现在市场在动哪些币，主流/热门/机会分别是谁。
2. **结构化信号感知**：哪些币出现放量、突破、趋势、偏离、极端涨跌。
3. **Agent 观点感知**：Alpha/Beta/Gamma 分别盯什么币、为什么、什么条件改变判断。

一句话目标：

> 用户打开页面 30 秒内知道市场热点，停留 3 分钟能看到变化，停留 10 分钟能感到 Agent 在持续观察市场。

---

## 3. 信息分层

### 3.1 行情层：30s 刷新

低成本、常刷新、视觉上始终可见。

展示对象来自 Task 08：

- 主流：BTC / ETH / SOL / USDT
- 热门：CoinGecko trending top 3
- 机会：24h 涨幅 top 2 + 跌幅 top 1

页面表现：

- CoinTickerStrip 三组横向展示。
- 每个 chip 显示 symbol、price、24h change。
- 只主流 4 币支持 modal；热门/机会币先不弹 modal。
- 标记数据源与降级状态。

价值：用户一眼知道“今天市场看哪里”。

### 3.2 信号层：60s 刷新

低成本计算层，不依赖 LLM。

信号候选：

- 5m / 15m 涨跌幅异常。
- 24h 强弱排序变化。
- 接近前高 / 前低 / 整数关口。
- volume ratio 放大。
- EMA12/13、EMA144/169 排列状态。
- 布林带/均线偏离或 RSI 极端值（如果数据可得）。
- 主流币与热门/机会币之间的轮动。

页面表现建议：

- 新增 `MarketEventFeed`，穿插在 Agent 消息流上方或右侧。
- 每条事件短句化，例如：
  - `SOL 5m 放量 2.1x，接近 154.2 前高`
  - `BTC 15m EMA12 重新站上 EMA13`
  - `DOGE 24h +18.4%，进入机会池`
- 事件流只展示最近 8-12 条，避免刷屏。

价值：即使 LLM 没刷新，页面也有持续变化。

### 3.3 Agent 解读层：3-5min 刷新 + 事件触发

高成本 LLM 层，负责把行情和信号变成“人格化观点”。

建议频率：

- 默认 3 分钟一轮，而不是完全靠前端 60s 请求。
- 页面后台停止刷新。
- 若出现强事件，可提前触发一轮，但需最小间隔，例如 90s。
- 若 LLM 失败，继续展示上一轮观点 + 新信号流，不让页面空转。

每轮内容：

- Alpha：突破派，优先看热门/机会币是否接近关键位。
- Beta：趋势派，优先看主流/热门币的 EMA 和趋势延续。
- Gamma：回归派，优先看机会/主流币的偏离、超买/超卖。

价值：Agent 负责“解释为什么重要”，不是承担所有刷新。

## 4. 页面结构建议

### 4.1 顶部：多币种 ticker

使用 Task 08 的三组 ticker：

- 主流
- 热门
- 机会

这一层是用户进入页面的第一价值信号。

### 4.2 中部：Agent 状态 + 持久观点卡

新增三个 `AgentFocusCard`：

- 当前关注币
- 当前判断
- 触发条件
- 失效条件
- 更新时间

示例：

```text
Alpha · 突破派
关注：SOL
判断：154.2 前高附近缩量，先防假突破。
触发：放量站上 154.2 后回踩不破。
失效：跌回 150 下方。
```

价值：用户不用翻聊天历史，也知道三个 Agent 当前分别在盯什么。

### 4.3 下部：消息流 + 市场事件流

保留消息流，但不再只依赖三条 Agent 消息。

建议展示顺序：

1. 市场事件自动插入。
2. Agent 轮询输出插入。
3. 用户回来时显示“有新内容”提示。
4. 历史仍可向上滚动。

消息类型建议扩展：

```ts
type WatchFeedItem =
  | { type: "agent"; agentId: AgentId; content: string; timestamp: number }
  | { type: "market-event"; symbol: string; severity: "info" | "hot" | "risk"; content: string; timestamp: number };
```

价值：页面更像实时看盘室，而不是聊天窗口。

## 5. 数据与成本策略

### 5.1 刷新节奏

| 层级 | 刷新 | 成本 | 说明 |
|---|---:|---:|---|
| Ticker | 30s | 低 | CoinGecko + cache |
| Market signals | 60s | 低 | 本地计算 |
| Agent LLM | 3-5min | 中 | DeepSeek primary |
| Event-trigger LLM | 最小间隔 90s | 中 | 只在强信号触发 |
| History | 初次加载 + 新内容提示 | 低 | 后端 buffer |

### 5.2 LLM 不应承担的工作

- 不负责发现所有市场事件。
- 不负责高频刷新。
- 不负责展示全部 ticker。
- 不负责可机械计算的强弱排序。

LLM 只负责：

- 选择关注对象。
- 用 Agent 人格解释信号。
- 给触发/失效条件。
- 形成可读观点。

Task 08 解决：

- 10 币数据源。

- LLM 从 10 币中选币。

Task 09 解决：

- 用户停留时看什么。
- Agent 观点如何持久展示。
- 信号如何在 LLM 刷新间隙继续产生价值。
- 页面从“消息流”升级为“行情旁观台”。

建议落地顺序：

1. Task 08：多币种池。
2. Task 09A：MarketEventFeed + 低成本信号层。
3. Task 09B：AgentFocusCard + Agent 当前关注状态。
4. Task 09C：刷新策略优化，加入事件触发和最小间隔。

## 7. 验收标准

### 7.1 用户价值

- 打开页面 30 秒内能看到 10 个币的市场分布。
- 停留 3 分钟内至少看到 ticker 或 market event 更新。
- 停留 10 分钟能看到至少 2-3 轮 Agent 观点或事件触发更新。
- 三个 Agent 不总是说同一个币。
- 用户能一眼看到 Alpha/Beta/Gamma 当前关注点，而不是翻历史。

### 7.2 信息质量

- Agent 每条观点必须包含 symbol、数字、条件。
- Market event 必须来自可解释数据，不编造。
- 数据源降级时清晰标记。
- 热门/机会币不进入 modal，避免过早扩成本。

### 7.3 成本与稳定性

- LLM 默认不高于 3 分钟一轮。
- 页面后台停止轮询。
- CoinGecko / CoinW 任一失败时，页面仍可显示主信息。
- LLM 失败时使用上一轮观点 + market events 承接体验。

## 9. 当前建议

当前最稳妥路径：

1. 先照 Task 08 做多币种池，让页面信息密度立刻提升。
2. 不急着提高 LLM 频率，先加低成本 MarketEventFeed。
3. 再做 AgentFocusCard，把三位 Agent 的“当前关注”固定住。
4. 最后再考虑事件触发 LLM。

原因：用户价值感来自“持续有东西发生”，不一定来自“LLM 更频繁说话”。
